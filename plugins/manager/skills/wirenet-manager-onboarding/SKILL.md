# WireNet Manager Onboarding

Make the first meeting feel like a capable work companion learning the user's
world, not a setup wizard. Preserve Jason Liu's onboarding sequence while using
WireNet's explicit source and write approvals.

Read `references/first-meeting-flow.md` completely before starting.
Read `references/manager-task-template.md` before proposing durable instructions,
renaming, pinning, or a recurring check-in for the long-running Manager task.

## Setup State

Classify quietly:

- `brand_new`: healthy Manager exists, but no calibrated work map or continuity
  setup exists. Run the full first meeting.
- `partial`: some map, sources, memory, writing profile, task, or check-in state
  exists. Fill only the gaps.
- `established`: the Manager already carries a useful map. Skip onboarding and
  use `$wirenet-manager` for the request.

## Full Flow

1. Start with the language-specific exact hello in the reference.
2. Build a grounded first map from the current task and local Manager context.
   Inspect source capability state, but do not read connected communication or
   cloud content without approval.
3. Interview one open question at a time for corrections, active and ignored
   work, near-term importance, dropped-ball patterns, important people, project
   locations, and missing source capabilities. Update the map after each answer.
4. When connected evidence would materially improve the map, explain the
   proposed scope and ask before a targeted read. Use the result to refine the
   interview and map rather than producing a generic source digest.
5. After calibration, propose the specific Manager concepts and runtime routes
   that future work needs. Ask before writing inferred durable content. For
   approved project roots, use the bootstrap discovery helpers, then delegate
   classification and packet creation to `$wirenet-manager-project`. Delegate
   canonical collaborator concepts to `$wirenet-manager-person`. Preview the
   global core guidance and optional user-owned routing only when useful.
6. Propose one core Manager check-in and what it will watch.
7. Offer narrow project, people, or daily monitor tasks only when they reduce
   noise; suggest 09:00 and 16:00 in the user's timezone as the default daily
   checkpoints.
8. When approved sent email or Slack evidence is sufficient, offer
   `$write-like-me-bootstrap` with the concrete wording in the reference.
9. Explain that the local Manager is the user's reviewable shared memory, offer
   to rename or pin the current task as the Manager home, and close with the
   reference's recap structure.

## Approval Gates

Ask separately before installing or connecting plugins, reading connected
sources, writing inferred Manager knowledge, generating a personal skill,
creating or changing tasks, or creating automations. Never treat approval for
one step as approval for the next.

## Done Means

Onboarding is complete only when the map, interview, source gaps, durable-file
proposals, core check-in, optional monitor offer, write-like-me offer, shared
memory explanation, task guidance, and recap are handled, declined, unavailable,
or explicitly deferred.
